# MVP 技术方案与避坑计划

## Summary
- MVP 主技术栈采用 `Godot + GDScript`，不使用纯 C++ 作为主开发语言。
- 内容工作流采用 `CSV 表驱动 + 自动导入生成游戏内数据资源`，保证后续平衡、扩内容和批量迭代可控。
- 战斗采用 `纯数据模拟核心 + 日志协议驱动表现层`，表现层只消费规则结果，不允许反向驱动规则。
- 规则层从第一天开始支持 `多随机流隔离 + 自动测试 + seed 复现`，这是避免 STS 类项目后期失控的关键。

## Key Technical Decisions
### 1. 主技术栈
- `Godot` 负责工程框架、UI、场景、流程控制、资源组织和导入管线。
- `GDScript` 负责绝大多数 MVP 逻辑，优先换取迭代速度。
- `C++ / GDExtension` 只作为后续热点优化手段，不进入 MVP 主流程设计。
- 默认首发平台按 `PC` 设计，输入和 UI 先以键鼠为主，手柄只做结构预留。

### 2. 内容层架构
- 所有核心内容先按表驱动设计，至少包括：
  - NPC 定义
  - 骰子定义
  - 敌人定义
  - 状态效果定义
  - Run 节点与奖励定义
  - 关系节点与解锁条件
- MVP 的主表格式用 `CSV`。
- 导入层负责把 CSV 转成 Godot 内部可消费的数据对象，运行时不直接依赖原始表结构。
- 导入流程优先做成 `EditorPlugin` 或等价编辑器导入机制，支持 CSV 变更后自动重建 `.tres` / 内部资源。
- 目标是形成“改表 -> 保存 -> Godot 内资源更新”的闭环，避免手工导入造成版本和数值错位。
- 复杂文本或分支事件如果 CSV 明显不适配，可后续补 `JSON`，但 MVP 先不引入混合格式复杂度。

### 3. 规则层与表现层分离
- 战斗规则层必须是 `纯数据模拟核心`：
  - 输入：战斗状态、角色数据、敌人数据、随机种子、玩家决策
  - 输出：状态变化、行动结果、结算结果、结构化日志
- 表现层只负责：
  - 展示当前状态
  - 收集玩家输入
  - 按日志协议回放结果
- 不允许把伤害、Buff、结算顺序、回合推进写在动画回调、按钮事件或场景节点副作用里。
- 必须定义统一 `ActionLog / BattleLog` 协议，作为规则层对 UI 的唯一“剧本”输出。
- UI 动画、数值飘字、镜头表现都以日志事件为准，确保表现与数据同步。

### 4. 随机与复现
- 从 MVP 开始统一使用带种子的随机系统。
- 必须隔离不同职责的随机流，至少拆分：
  - `rng_run`：地图节点、奖励、事件
  - `rng_dice`：投骰与骰面相关结果
  - `rng_ai`：敌人行为分支或决策变体
- 各随机流由 Run 层 `master_seed` 派生，但运行中互不污染。
- 不允许“敌人 AI 决策”和“玩家掷骰”共用一个 RNG，否则玩家输入顺序会污染整个复现链。
- 每次 Run 必须能记录 seed 和关键决策，支持基本重放或复现。
- 调试日志中要能看到关键随机结果和其随机流来源。

### 5. 状态效果模型
- MVP 不做脚本化、自由拼装的万能 Buff 系统。
- 状态效果优先采用 `原子化定义`，例如：
  - `Trigger`
  - `OperationType`
  - `Value`
  - `Duration`
  - `StackRule`
  - `Source`
- 例如类似 `OnDealDamage + PowerMul + 1.5` 这样的标准化数据组合，而不是为每个 Buff 写单独逻辑脚本。
- 规则层通过统一循环解析状态效果，保证：
  - 性能可控
  - 行为可测试
  - 数值可查
  - 来源可追踪
- 复杂特例效果只在后期少量引入，且必须经过明确的扩展点设计，不进入 MVP 默认路径。

### 6. 数据结构预留
- 虽然 MVP 先做 `1 名 NPC 上场`，但这些结构必须按未来多角色扩展设计：
  - 出战单位容器
  - 行动顺序队列
  - 目标选择接口
  - 状态效果挂载
  - 骰池归属
  - 局内临时成长归属
- 主角数据和战斗角色数据必须从一开始拆开：
  - `主角数据`：关系、剧情、解锁、Hub 权限
  - `NPC战斗数据`：生命、骰池、状态、成长、装备、临时效果
- 这样后面从 `1人上场` 扩到 `2-3人上场` 时，不需要推倒战斗底层。

## Implementation Order
### 1. 规则底座
- 先搭建纯数据核心的基础模块：
  - 多随机流系统
  - 通用状态容器
  - 回合流程状态机
  - 行动解析
  - 结构化日志协议
- 同时确定 CSV 导入管线最小版本，至少能导入 NPC、骰子和敌人三类数据。
- 先不做 UI，只做最小调试输出验证规则层成立。
- 这一步必须先把 `BattleLog / ActionLog` 格式定下来，因为后续表现层要完全依赖它。

### 2. 单 NPC 战斗原型
- 用导入后的数据驱动首个可玩战斗：
  - 1 名 NPC
  - 2-3 种基础敌人
  - 攻击、防御、回复、基础状态
  - 胜负判定和战后结果
- 此阶段同时建立第一批自动测试：
  - 基础伤害结算
  - 防御减伤
  - 回复边界
  - 回合顺序
  - 相同 seed 结果一致
  - `rng_dice` 与 `rng_ai` 互不污染

### 3. 构筑与局内成长
- 在单战斗成立后，补上骰池替换、成长分支、局内强化。
- 导入表扩展到：
  - 状态效果
  - 奖励项
  - 临时成长项
- 状态效果全部优先走原子化数据定义，不直接写特例脚本。
- 自动测试补到：
  - 骰子替换后打法变化
  - Buff / Debuff 叠加
  - 成长效果在后续回合中的持续性
  - 状态来源可追踪

### 4. Run 系统与撤离
- 接入路线图、节点状态、Run 级资源记录和撤离判定。
- Run 层先只支持：
  - 普通战斗
  - 事件
  - 补给
  - 撤离
- Run 层使用独立 `rng_run`，不得和战斗层随机混用。
- 测试重点放在：
  - 路线推进状态
  - 临时成长跨节点保留
  - 撤离与失败结算差异
  - 同 seed 的路线生成一致性

### 5. Hub 与局外成长
- Run 结算稳定后，再接最小 Hub。
- Hub 只做功能性版本：
  - 查看 NPC
  - 配置骰池
  - 消耗资源解锁成长
  - 再次进入 Run
- 导入表扩展到局外成长和解锁条件。
- 测试重点是 `Hub -> Run -> 结算 -> Hub` 的闭环一致性。

### 6. 关系与剧情最小实现
- 最后接入关系和剧情系统。
- 关系节点也走表驱动，至少支持：
  - 触发条件
  - 显示文本 / 事件 ID
  - 奖励内容
  - 解锁后续节点
- MVP 只做 1 条基础关系线和少量解锁，先验证“关系会反哺玩法”。
- 此阶段不做复杂剧情编辑器，避免过早投入工具开发。

## Public Interfaces / Contracts
- 需要尽早固定的核心接口：
  - `CombatState`
  - `CombatAction`
  - `CombatResolution`
  - `ActionLogEntry`
  - `RunState`
  - `RunSeedBundle`
  - `StatusEffectDefinition`
- `ActionLogEntry` 至少需要包含：
  - 事件类型
  - 事件时序
  - 行动发起者
  - 目标
  - 关联骰子或效果 ID
  - 数值结果
  - 来源信息
- 导入后的资源对象必须与原始 CSV 解耦，避免运行时直接依赖表字段名。
- 状态效果定义必须包含统一触发时机和叠层规则，防止后续同类 Buff 无法共存。

## Test Plan
- 规则层自动测试必须覆盖：
  - 伤害、防御、回复、死亡判定
  - 状态效果触发时机和持续回合
  - 骰子结果解析
  - 相同 seed 的结果一致
  - 不同随机流互不污染
  - 非法输入或空数据时的安全失败
- Run 层测试必须覆盖：
  - 路线生成复现
  - 节点推进
  - 撤离结算
  - 局内成长跨节点保留
- Hub 层测试必须覆盖：
  - 局外资源回流
  - 解锁条件判定
  - 成长后进入下一轮 Run 生效
- 关系层测试必须覆盖：
  - 条件触发
  - 奖励发放
  - 已解锁节点不重复错误触发
- 日志协议测试必须覆盖：
  - 同一规则输入输出相同日志序列
  - UI 可仅依赖日志回放战斗结果
  - 日志事件顺序与结算顺序一致

## Assumptions
- 默认 MVP 以 `玩法验证优先`，不优先投入复杂编辑器、复杂剧情工具或性能优化。
- 默认 Godot 版本选当前稳定版，不追新测试版特性。
- 默认 CSV 由手工维护或简单表格工具导出，不先建设专用内容管理后台。
- 默认 `日志协议` 和 `seed 记录` 属于 MVP 必做项，不视为后期调试附属品。
- 默认首版 UI 追求信息可读，不追求复杂动画和高表现演出。
