# GDD 2.0
## 项目暂名：TBD

## 1. 游戏概述

### 1.1 项目定位
本项目是一款以多角色养成和骰子构筑为核心的单机 Roguelike RPG。
玩家将以“主角”这一叙事与情感视角进入世界，组织并培养可出战的 NPC 队伍，在一次次 Run 中获取资源、承担风险、选择撤离或继续推进，并在局外推动角色关系、剧情与长期成长。

### 1.2 类型定义
- 单机 Roguelike
- Dice Building
- Party RPG
- Hub 养成 / 剧情推进
- Furry 同性向角色关系与恋爱叙事

### 1.3 核心卖点
- 以 NPC 为核心战斗单位的骰子构筑体验
- 基于 Run 推进与撤离判断的风险/收益博弈
- 通过主角视角推进 NPC 关系、剧情与恋爱内容
- 多角色成长方向与构筑分化带来的长期养成乐趣

### 1.4 目标玩家
- 喜欢《杀戮尖塔》式回合构筑玩法的玩家
- 喜欢《横跨方尖碑》式队伍养成与角色联动的玩家
- 喜欢单机 RPG、角色互动、关系推进和恋爱内容的玩家
- 愿意接受中度随机性和资源压力、同时追求长期成长反馈的玩家

### 1.5 核心体验目标
本项目希望提供三层核心体验：

1. 战斗层：通过骰子构筑和回合决策获得明确的策略反馈
2. Run 层：在推进、成长、受伤、资源消耗和撤离之间做风险判断
3. 情感层：通过主角与 NPC 的关系推进，建立长期投入与角色依恋

## 2. 核心玩法循环

### 2.1 主循环
Hub 大本营
→ 编队与构筑准备
→ 进入 Run
→ 分叉路线推进
→ 战斗 / 事件 / 补给 / 撤离判断
→ 撤离或完成 Run
→ 带回资源并进行局外成长
→ 推进剧情、关系与新内容解锁
→ 下一轮 Run

### 2.2 闭环作用
- Hub：提供管理、成长、关系和剧情反馈
- Run：提供即时战斗压力与资源获取
- 撤离：提供风险控制与收益取舍
- 局外成长：形成长期目标与持续解锁
- 骰子构筑：贯穿战斗内外，是所有系统的第一核心

### 2.3 闭环设计原则
- 玩家每一轮 Run 都应获得短期反馈和长期意义
- 玩家失败后也应获得有限前进动力，不应完全失去成长目标
- 关系推进必须与玩法收益相连，不能只停留在文本层面
- 撤离必须是“有代价但合理”的选择，而不是最佳固定策略

## 3. 核心角色结构

### 3.1 主角定位
主角是玩家在世界中的身份载体，是剧情和关系推进的核心人物，但不是战斗单位。
主角不直接参与攻击、防御或常规技能行动，不占用标准战斗角色位。

主角的核心作用包括：
- 玩家代入视角
- 队伍组织者与战术制定者
- 剧情推进核心
- NPC 关系与恋爱内容的主要承接者
- Hub 中重要决策与解锁的触发主体

### 3.2 NPC 定位
NPC 是实际参与战斗和成长构筑的主体。
每个 NPC 拥有独立的：
- 骰池
- 技能逻辑
- 成长分支
- 角色剧情
- 关系进度
- 构筑倾向

玩家的主要玩法表达，是通过培养 NPC 并塑造其最终打法来完成的。

### 3.3 角色设计方向
项目不采用严格锁死的战法牧结构。
每个角色都应具备基础的攻、防、回复能力，同时拥有更适合发展的方向。

设计目标是：
- 保证角色在早期可独立成立
- 避免职业绑定过强导致构筑过早僵化
- 让玩家通过构筑选择，而不是系统预设，决定角色最终定位

## 4. 系统模块拆解

### 4.1 Hub / 大本营
作用：
- 接取任务
- 与 NPC 互动
- 推进关系与剧情
- 管理队伍、资源、骰池与装备
- 进行局外成长和内容解锁

Hub 是项目的长期成长中心，也是玩家情感投入的核心场所。

### 4.2 NPC 构筑系统
作用：
- 定义角色战斗方式
- 支撑骰子成长和流派分化
- 构成战斗策略的核心来源

该系统是项目最重要的玩法模块，优先级高于装备和外围辅助系统。

### 4.3 Run 系统
作用：
- 提供战斗、事件和节点推进
- 承载局内成长和风险变化
- 形成资源获取与节奏控制
- 将玩家置于撤离与继续深入的抉择中

### 4.4 撤离系统
作用：
- 为 Run 提供风险/收益取舍
- 降低失败带来的完全挫败感
- 强化“见好就收”与“继续冒险”的博弈感

### 4.5 局外成长系统
作用：
- 承接 Run 收益
- 解锁角色成长内容
- 提供长期目标
- 强化重复游玩的动力

### 4.6 关系与剧情系统
作用：
- 增强角色塑造与玩家投入
- 解锁专属剧情、支线和恋爱内容
- 对玩法层提供骰子、被动、事件或分支收益

## 5. 核心玩法机制

### 5.1 骰子构筑系统
骰子构筑是项目的第一核心。
角色通过独立骰池表达其战斗风格、成长方向与策略价值。

骰子可按功能分为：
- 攻击类
- 防御类
- 回复类
- 功能类
- 特殊机制类

构筑目标包括：
- 形成清晰的角色战斗差异
- 让玩家通过成长和选择塑造流派
- 在局内成长和局外成长之间建立衔接
- 保证构筑乐趣来源于“打法变化”，而不是单纯数值增大

设计原则：
- 骰子必须是战斗系统的绝对核心
- 装备、道具、关系收益都只能辅助构筑，不应取代构筑
- 角色成长应体现“方向变化”，而非仅仅是线性强化

### 5.2 战斗系统
战斗采用 STS 式回合制表现，强调信息可读性、决策明确性和节奏效率。

战斗设计目标：
- 玩家能够明确理解角色当前能力与敌方意图
- 决策围绕骰子资源、技能选择和局势判断展开
- 构筑变化应在战斗中产生明显手感差异
- 演出服务角色塑造，但不拖慢回合体验

MVP 阶段重点：
- 先验证单角色战斗是否完整成立
- 确保一个角色也能形成完整决策循环
- 后续再扩展协同、支援和多角色联动机制

### 5.3 队伍系统
项目长期目标是实现多人上场与多角色联动。
但在 MVP 阶段，优先验证单 NPC 上场的完整性。

发展路径：
- 第一阶段：单 NPC 上场
- 第二阶段：双人协同
- 第三阶段：三人队伍联动

设计原则：
- 当前玩法先围绕“单角色构筑成立”展开
- 所有底层结构需为未来 2 人、3 人上场预留扩展空间
- 多角色系统建立在单角色体验稳定之后

### 5.4 Run 结构
Run 采用分叉路线图形式推进。
玩家在不同节点间做路径选择，决定资源节奏、成长效率和风险高低。

节点类型包括但不限于：
- 战斗
- 事件
- 补给
- 商店
- 精英
- 撤离
- Boss

Run 的设计目标：
- 提供清晰的前进压力
- 让构筑在局内逐步成型
- 让路线选择成为资源管理的一部分
- 让玩家始终处于“继续推进是否值得”的心理判断中

### 5.5 局内成长
在一次 Run 中，玩家可获得临时成长内容，例如：
- 骰子强化
- 临时被动
- 一次性道具
- 特殊事件奖励
- 构筑修正项

作用：
- 加强单局变化感
- 让构筑更快产生辨识度
- 提供即时反馈和成型快感

### 5.6 撤离系统
撤离系统是第二核心机制。
它的目标是在不破坏主玩法重心的前提下，为 Run 增加心理张力和资源取舍。

基础原则：
- 玩家在特定节点获得撤离机会
- 撤离可稳定带回部分资源
- 完整完成 Run 的收益高于撤离
- 撤离收益高于失败空手而归
- 玩家必须能直观看懂撤离与继续推进的代价差异

MVP 阶段：
- 采用简洁、易理解的撤离收益结构
- 暂不优先投入复杂耐久和损坏规则
- 先验证撤离是否真正形成决策价值

### 5.7 局外成长
局外成长负责承接资源、解锁内容、建立长期目标。

主要成长内容：
- 新骰子解锁
- 角色成长分支开启
- 装备解锁与强化
- 新任务或区域开放
- 新剧情事件解锁
- 关系等级提升

设计目标：
- 让每轮 Run 都对长期成长有贡献
- 让玩家持续投入特定角色和构筑方向
- 让失败也保留部分前进价值

### 5.8 关系与剧情系统
关系系统是本项目的重要情感与商业化识别层之一。
它不仅承担角色塑造功能，也必须对玩法形成明确反馈。

关系系统提供：
- NPC 个人剧情
- 支线事件
- 恋爱线推进
- 专属奖励或成长分支解锁
- 构筑相关的专属收益

设计原则：
- 关系推进必须影响玩法
- 剧情内容要和角色战斗风格、成长方向一致
- 主角在关系系统中始终是核心连接点

## 6. 核心资源设计

### 6.1 骰子
角色战斗和构筑的核心资源。
用于定义角色的能力表达、构筑方向和成长深度。

### 6.2 装备
为角色提供辅助性修正。
装备的设计目标是增强策略空间，而不是取代骰子系统本身。

### 6.3 道具
主要用于局内临时收益、恢复、修正战局和制造节奏变化。

### 6.4 货币 / 材料
用于局外购买、升级、解锁和成长。
是长期推进的基础资源。

### 6.5 关系值
记录主角与 NPC 的关系进度。
应同时服务于剧情解锁和玩法成长，不应只是叙事指标。

### 6.6 剧情 / 任务解锁
用于推进主线、支线、角色专属内容和后续玩法区域。

### 6.7 资源设计原则
- 骰子为主，其他资源为辅
- 资源结构必须服务长期成长循环
- 关系资源必须具备玩法反馈
- 装备与道具不应破坏构筑主导权

## 7. 核心设计原则

### 7.1 骰子构筑优先
所有战斗与成长设计，都应以构筑是否成立、是否有趣、是否可成长为最高判断标准。

### 7.2 主角不参战
主角的核心价值在叙事、关系、组织和代入感，而不在战斗位表现。
不应为了参与感而破坏整体结构。

### 7.3 NPC 培养是主要玩法表达
玩家的战斗表达、长期投入和成就感，应主要来自 NPC 培养与队伍构筑。

### 7.4 关系推进必须反哺玩法
剧情、互动和恋爱内容必须与成长、构筑或解锁形成实际关联。

### 7.5 撤离系统提供张力而非喧宾夺主
撤离很重要，但不应压过骰子构筑本身成为项目最复杂系统。

### 7.6 先做成立，再做扩展
MVP 先验证单角色构筑和 Hub-Run-Hub 闭环。
多人协同、复杂联动和更深层资源模型在此之后逐步加入。

## 8. MVP 开发路线

### 8.1 MVP 目标
构建一套完整的垂直切片，验证以下问题：
- 单 NPC 战斗是否成立
- 骰子构筑是否足够有趣
- Hub 与 Run 的闭环是否清晰
- 撤离是否具备决策价值
- 关系推进是否能驱动玩家继续投入

### 8.2 MVP 内容范围
- 1 名可完整出战的 NPC
- 2 至 3 名可培养 NPC
- 基础 Hub 系统
- 基础分叉路线图 Run
- 若干基础敌人和事件
- 基础撤离结算
- 基础关系推进内容
- 基础局外成长和骰池扩展

### 8.3 MVP 非优先项
- 复杂耐久与损坏机制
- 主角战斗内独立系统
- 高复杂度多人组合技
- 过多角色数量
- 复杂经济与资源公式
- 大量剧情分支并行展开

### 8.4 MVP 验证标准
- 玩家能在一次完整流程中理解产品核心循环
- 单角色构筑能产生明显打法差异
- 撤离决策具备真实张力
- 关系推进能提供明确情感和玩法回报
- 玩家会对下一轮 Run 产生明确动机

## 9. 项目结论
本项目的本质，是一款围绕 NPC 队伍构筑展开的单机骰子 RPG。
其核心玩法不是主角亲自战斗，而是玩家通过主角这一身份，去组织、培养、理解并投入到可出战的 NPC 队伍之中。

项目的系统优先级明确如下：
1. 骰子构筑
2. Run 风险与撤离决策
3. 局外成长
4. 关系与恋爱推进
5. 装备与其他辅助系统

MVP 的首要任务不是铺开大量内容，而是先验证以下事实：
一个角色是否已经足够好玩，一次 Run 是否已经足够成立，一个关系推进是否已经足够让玩家投入。
